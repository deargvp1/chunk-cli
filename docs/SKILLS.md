# Skills

Skills are instruction files for AI coding agents. Installing them teaches agents how to use `chunk` commands — code review, remote validation, test gap analysis, and CI debugging — through natural-language triggers.

## Installing skills

```bash
chunk skill install     # install or update all skills
chunk skill list        # show installation status per agent
```

Skills are installed into the configuration directories of whichever agents are present on your machine:

| Agent | Install path |
|---|---|
| Claude Code (CLI / desktop) | `~/.claude/skills/` |
| Cursor | `~/.cursor/skills/` |
| Codex | `~/.codex/skills/` |

`chunk skill list` shows the status for each skill on each agent:

- **current** — installed and up to date
- **outdated** — installed but a newer version is available; re-run `chunk skill install`
- **missing** — not installed; run `chunk skill install`
- **n/a** — agent not found on this machine

Skills are versioned. Run `chunk skill install` again after upgrading `chunk` to keep them in sync.

## Available skills

### chunk-review

**Trigger phrases**: "review my changes", "chunk review", "review my diff", "review this PR", "review PR #123", "review PR `<url>`"

Applies your team's generated review standards to the current diff. The review runs in an isolated subagent so prior conversation context does not influence findings.

**Prerequisites**:
- `.chunk/context/review-prompt.md` must exist (generated by `chunk build-prompt`)
- For PR reviews: `gh` CLI must be installed and authenticated

**What it does**:

1. Determines the diff scope — PR diff via `gh`, staged changes, or uncommitted changes against main
2. Loads `.chunk/context/review-prompt.md` from the PR branch (or local fallback)
3. Runs a two-pass review in a subagent:
   - Pass 1: comprehensive scan against all review standards
   - Pass 2: validate each finding with a concrete failure scenario; filter to Critical/High issues; cap at 10 comments
4. Returns filtered findings with file, line number, impact rating, and suggested fix

**Example output**:

```
## Review Summary

Files reviewed: 4
Issues identified: 2

### 1. Missing error check on file write
`internal/store/write.go:42` | **High**

`os.WriteFile` can fail silently if the destination is read-only or the disk
is full. The error return is discarded here. Downstream code assumes the file
exists and will panic on next read.

Suggested fix: check and return the error.
```

If no issues survive filtering, the skill reports "No significant issues identified."

### chunk-sidecar

**Trigger phrases**: "validate on the sidecar", "run tests on the sidecar", "sync to sidecar", "sidecar dev loop", "validate remotely"

Runs the sync → validate loop on a remote CircleCI sidecar. Also handles one-time sidecar creation, environment setup, and snapshotting.

**Prerequisites**:
- CircleCI token configured (`chunk auth set circleci`)
- An active sidecar (`chunk sidecar use <id>`)

**What it does**:

1. Checks auth status (`chunk auth status`)
2. Finds the active sidecar (`chunk sidecar current`)
3. Syncs local changes to the sidecar (`chunk sidecar sync`)
4. Runs validation commands remotely (`chunk validate --remote`)
5. Reports results; maps error paths back to local files

If there is no active sidecar, the skill prompts you to select or create one before proceeding.

**The dev loop in practice**:

```
You: validate on the sidecar
Agent: syncing local changes...
       running chunk validate --remote...
       ✗ test failed: internal/store_test.go:88: assertion failed
       Fixing the issue...
       syncing again...
       running chunk validate --remote...
       ✓ all validations passed
```

**Parallel sessions**: When multiple Claude sessions are open in the same repo, each session automatically targets its own sidecar. State is keyed by `CLAUDE_SESSION_ID` and stored in `XDG_DATA_HOME` (see Architecture docs). No manual configuration is needed.

### chunk-testing-gaps

**Trigger phrases**: "find testing gaps", "chunk testing-gaps", "mutation test", "mutate this code", "test mutation coverage", "find surviving mutants"

Runs a four-stage mutation testing process to find undertested code paths.

**What it does**:

1. **Discovery** — identifies mutation candidates in changed code
2. **Validation** — verifies surviving mutants using git worktrees and CI
3. **Production cross-reference** — checks whether survivors touch production code paths
4. **Risk assessment** — reports high-risk survivors with recommended tests

---

### debug-ci-failures

**Trigger phrases**: "debug CI", "why is CI failing", "fix CI failures", "find flaky tests", "what broke in CI", "check CircleCI"

Analyzes CircleCI pipeline failures and identifies flaky tests.

**What it does**:

1. Detects the project and current pipeline
2. Fetches build failure logs
3. Analyzes test results for patterns
4. Distinguishes genuine failures from flaky tests
5. Reports root cause with remediation steps

## Keeping skills up to date

Skills are bundled into the `chunk` binary. When you upgrade `chunk`, run `chunk skill install` again to push the updated skill files to your agents.

```bash
chunk upgrade           # update chunk itself
chunk skill install     # update skill files in agent dirs
```

---

## Using skills without an agent

Skills are markdown files the agent reads. You can read them directly for reference:

```bash
# Claude Code skill files after install
cat ~/.claude/skills/chunk-review.md
cat ~/.claude/skills/chunk-sidecar.md
```

## Troubleshooting

**`chunk skill list` shows "n/a" for all agents**

The agent configuration directories were not found. Make sure the agent is installed. Claude Code creates `~/.claude/` on first launch.

**Skills show "outdated"**

Run `chunk skill install` to update them.

**Agent doesn't respond to skill triggers**

Confirm the skill is installed (`chunk skill list`). Some agents require a restart after new skill files are added. Check the agent's documentation for how it loads skill files.

**`chunk-review` says "review-prompt.md not found"**

The team review prompt hasn't been generated yet. Run `chunk build-prompt` to create it, then commit `.chunk/context/review-prompt.md` so all team members have it.
